## 1. Company-service (업체 및 직원 관리)
- 해당 서비스는 시스템 내의 업체(Company)를 등록하고, 해당 업체에 소속된 직원(CompanyMember)들을 제어하는 역할을 담당합니다. Microservice Architecture(MSA) 환경에서 유저 서비스와의 정밀한 데이터 동기화를 위해 Feign Client 통신을 적극적으로 활용합니다.

### Company (업체)
- 업체 등록

  - 권한: 허브 매니저 혹은 마스터 권한을 가진 유저만 등록이 가능합니다.

  - 동작 방식: 요청(Request)으로 들어온 사업자 번호를 기반으로, 유저 서비스에 Feign 통신을 날려 해당 사업자 번호와 일치하는 유저의 ID를 받아옵니다. 이 받아온 ID를 해당 업체의 초기 마스터 멤버로 지정합니다.

- 업체 삭제

  - 권한: 허브 매니저 혹은 마스터만 가능합니다.

  - 동작 방식: 업체를 삭제할 때, 유저 서비스에 Feign 통신을 요청하여 해당 업체 ID를 바라보고 있는 모든 유저들의 company_id 필드를 전부 초기화(Null) 처리하거나 연쇄 조치를 취합니다.

### CompanyMember (업체 직원)
- 멤버 등록

  - 권한: 이미 등록된 업체 매니저만 본인의 업체에 직원을 추가할 수 있습니다.

  - 검증 및 방어 로직: Feign 통신을 통해 등록할 유저의 정보를 받아올 때, 해당 유저가 이미 다른 허브(hub_id)나 다른 업체(company_id)에 소속되어 있는지 철저하게 검증합니다. 이미 소속이 존재하는 유저라면 예외(Error)를 반환하여 소속 중복을 원천 차단합니다.

- 멤버 삭제

  - 권한: 허브 매니저 혹은 마스터만 가능합니다.

  - 동작 방식: 소속 직원을 삭제(해제)할 때, 유저 서비스에 Feign 통신을 보내 해당 유저의 company_id 연결을 끊어달라고 요청합니다.

## 2. Product-service (상품 및 하위 도메인 구조)
- 상품 서비스는 이커머스 및 물류 시스템의 핵심으로, 하나의 상품이 가질 수 있는 수많은 옵션의 형태와 최종 구매 단위인 베리언트(Variant)를 객체지향적 구조로 관리합니다.

```
Product (상품 본체)
├── List<ProductOption> (옵션 대분류: 사이즈, 색상)
│     └── List<ProductOptionValue> (옵션 값: 260, 블랙)
└── List<ProductVariant> (최종 조합/재고 단위: 블랙/260)
└── List<ProductVariantOption> (중간 매핑 테이블 - 행정병)
```

### 엔티티별 역할 및 핵심 설계 철학

- Product (상품)

  - 상품 도메인의 최상위 Root Aggregate입니다. 상품명, 설명, 기본 가격을 가집니다.

- ProductOption (옵션 대분류)

  - 상품과 옵션값 사이를 중재하며, 옵션의 이름(예: 사이즈, 색상)을 가집니다.

  - **[설계 철학 - 상태 및 수정 기능 제거]**: 옵션 대분류는 화면 조회 시 단순히 옵션 값들을 묶어주는 '껍데기(타이틀 명)' 역할만 수행합니다. 실제 소비자가 마주하는 품절 및 판매 제어는 하위의 '옵션 값'이나 '베리언트'에서 이루어지기 때문에, ProductOption에는 불필요한 상태(status) 필드를 과감히 제거하고 수정이 불가능하도록 미니멀하게 설계하여 성능과 데이터 정밀도를 높였습니다.

- ProductOptionValue (옵션 값 - 소대장)

  - 실제 옵션의 알맹이 데이터(예: 블랙, 260)와 해당 옵션 선택 시 기본가에 추가되는 추가 금액(additionalPrice)을 관리합니다.

  - 판매 중지(INACTIVE) 및 재개(ACTIVE) 상태를 가집니다.

- ProductVariant (옵션 조합/베리언트 - 중대장)

  - 물류 및 재고 조회 시 호출되는 최종 판매 단위입니다. 상품명과 옵션값들이 조립된 최종 이름(예: 나이키 에어포스 (블랙 / 260))과 최종 가격(기본가 + 추가 금액의 합)을 가집니다.

- ProductVariantOption (조합 매핑 엔티티 - 행정병)

  - ProductVariant와 ProductOptionValue 사이의 다리 역할을 하는 다대다(N:M) 해소용 중간 테이블입니다.

### 3. 핵심 기능: 재귀 기반의 자동 상품 등록 프로세스
상품을 처음 등록할 때, 관리자가 일일이 수십 개의 옵션 조합을 손으로 입력하는 것은 비효율적이며 데이터 오류를 야기합니다. 이를 해결하기 위해 ProductService의 registerProduct 메서드는 재귀(Recursion) 알고리즘을 활용한 복합 옵션 빌더 구조로 구현되었습니다.

-  동작 메커니즘 (How it works)
  1. 최상위 부모 생성: 입력받은 상품 기본 정보로 Product 엔티티를 영속화합니다.

  2. 옵션 트리 빌드: 요청으로 들어온 옵션 그룹(ex: 색상:[블랙, 화이트], 사이즈:[260, 270])을 순회하며 ProductOption과 ProductOptionValue를 먼저 데이터베이스에 안착시킵니다.

  3. 재귀적 조합 생성 (Combination):

     - 생성된 옵션 그룹 리스트의 depth를 타고 들어가며 수학적 카테시안 곱(Cartesian Product) 연산을 재귀 함수로 돌립니다.

     - [블랙, 화이트] 와 [260, 270] 이 들어온다면 재귀 스택을 통해 아래와 같이 총 4개의 조합을 유연하게 도출합니다.

       - 블랙 -> 260 픽 (조합 1 완료)

       - 블랙 -> 270 픽 (조합 2 완료)

       - 화이트 -> 260 픽 (조합 3 완료)

       - 화이트 -> 270 픽 (조합 4 완료)

  4. 베리언트 자동 생성 및 돈 계산:

     - 추출된 조합을 바탕으로 ProductVariant를 자동으로 생성합니다.

     - 이때, 상품 본체의 기본 가격과 재귀를 타고 내려오면서 합산된 옵션값들의 additionalPrice를 자동으로 합산하여 최종 베리언트 판매 가격(variantPrice)을 스스로 계산해 냅니다.

     - 이름 또한 상품명 + (옵션값 / 옵션값) 형태로 자동 빌드되어 ProductVariantOption(행정병) 매핑 테이블과 함께 안전하게 저장됩니다.

## 재귀 방식을 선택한 이유 (Why Recursion?)
만약 옵션이 2개(색상, 사이즈)로 고정되어 있다면 단순한 2중 for문으로 해결할 수 있습니다. 하지만 어떤 상품은 옵션이 없을 수도 있고, 어떤 상품은 옵션이 4개(색상, 사이즈, 기장, 로고유무)가 될 수도 있습니다.

재귀 알고리즘을 채택함으로써 옵션의 개수가 몇 개가 들어오든 코드 한 줄 수정 없이 동적으로 모든 N차원 옵션 조합을 완벽하게 계산하고 베리언트를 뽑아낼 수 있는 확장성을 확보했습니다.
