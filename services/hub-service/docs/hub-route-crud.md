# HubRoute CRUD API 및 캐싱 구현 (Step 7)

## 1. 구현 내용
- **HubRouteService**: 허브 간 이동 경로의 생성, 조회, 수정, 삭제 로직을 구현했습니다.
- **HubRouteController**: 경로 관리 API 엔드포인트를 제공하며, 권한 체크 및 페이지네이션 검증을 수행합니다.
- **캐시 연쇄 무효화**: 경로 데이터가 변경(`create`, `update`, `delete`)될 때, 해당 허브가 포함된 모든 최단 경로 탐색 결과(`path:*`) 캐시를 연쇄적으로 삭제하도록 구현했습니다.

## 2. 왜 이 방식을 선택했는가 (설계 의도)
- **데이터 정합성 유지**: `HubRoute`는 다익스트라 알고리즘의 원천 데이터이므로, 특정 구간의 정보가 변하면 네트워크 전체의 최단 경로 결과가 바뀔 수 있습니다. 따라서 경로 데이터 변경 시 모든 `path:*` 캐시를 일괄 삭제하여 정합성을 보장했습니다.
- **캐시 삭제 최적화**: `redisTemplate.keys()`의 성능 문제를 방지하기 위해 `RedisConnection` 수준에서 원시 바이트 기반의 일괄 삭제를 수행했습니다. (소규모 데이터셋이므로 SCAN보다 정합성 우선)
- **지연 로딩 대응**: `HubRouteResponse` 생성 시 연관된 `Hub` 엔티티의 ID와 이름을 안전하게 추출하도록 구성했습니다.

## 3. API 스펙
### POST /api/v1/hub-routes
- MASTER/HUB_MANAGER 전용
- 두 허브 간의 거리와 시간을 설정하여 새로운 간선(Edge) 생성
### GET /api/v1/hub-routes
- 전역 사용자 조회 가능
- 페이지 크기 제약(10, 30, 50) 적용

## 4. 주의사항 및 한계
- `redisTemplate.keys()` 작업은 Redis 성능에 영향을 줄 수 있으므로, 향후 데이터량이 방대해질 경우 별도의 인덱싱 전략이 필요할 수 있습니다.

## 5. 팀원 공유 사항
- 특정 허브 간의 소요 시간을 수정하면, 해당 구간을 지나는 모든 예약된 경로 안내에 영향을 줄 수 있으므로 주의가 필요합니다.
