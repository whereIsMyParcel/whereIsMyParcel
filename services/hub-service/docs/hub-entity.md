#  Hub 엔티티 및 Repository 구현 (Step 2)

## 1. 구현 내용
- **Hub 엔티티**: 전국 17개 물류 허브를 관리하기 위한 엔티티를 구현했습니다. UUID PK, 위경도 좌표, 주소 정보를 포함하며 JPA Audit 및 Soft Delete가 적용되었습니다.
- **HubRepository**: 기본 CRUD 기능과 데이터 초기화 시 중복 방지를 위한 존재 여부 확인 메서드를 포함합니다.
- **HubDataInitializer**: `ApplicationRunner`를 구현하여 서버 기동 시 전국 17개 주요 허브 데이터를 자동으로 DB에 적재합니다.

## 2. 왜 이 방식을 선택했는가 (설계 의도)
- **불변성 및 캡슐화**: `@Builder`를 `private` 생성자에 선언하고 외부에서는 `create()` 정적 팩토리 메서드만 사용하도록 제한하여 엔티티 생성 로직을 캡슐화했습니다.
- **좌표 유효성 검증**: 엔티티 내부에 위도(-90~90) 및 경도(-180~180) 범위를 검증하는 로직을 추가하여 잘못된 데이터 유입을 원천 차단했습니다.
- **Soft Delete**: `@SQLRestriction`을 통해 모든 조회 쿼리에서 삭제된 허브가 자동으로 제외되도록 설정했습니다.

## 3. API 스펙 (Request/Response DTO)
### CreateHubRequest (record)
- `name`: 허브 이름
- `address`: 허브 주소
- `latitude`: 위도 (-90 ~ 90)
- `longitude`: 경도 (-180 ~ 180)

## 4. 주의사항 및 한계
- 현재 데이터 초기화 시 `name` 기준으로 중복 체크를 하지 않고 전체 카운트가 0일 때만 실행됩니다. 추후 정교한 마이그레이션이 필요할 수 있습니다.

## 5. 팀원 공유 사항
- 허브 데이터는 서버 시작 시 자동으로 생성되므로 별도의 INSERT 쿼리 없이 테스트 가능합니다.
